# Prompt 04: CVRO Operation Implementations

You are implementing the four CVRO reasoning operations for the pilot pipeline.

This stage defines and implements the operations that can be forced from a saved `ReasoningState`:

1. `stop`
2. `continue`
3. `branch`
4. `verify_check`

Each operation must take a saved state as input and produce a standardized operation result that can later be scored and stored as a `CounterfactualOutcome`.

Do not implement counterfactual collection orchestration, controller training, baseline evaluation, or final reporting in this prompt. Focus on clean operation execution contracts, prompt construction, operation-specific constraints, and operation result schemas.

## Context

CVRO estimates the counterfactual value of different operations by forcing each operation from the **same saved intermediate reasoning state**.

The operations must be meaningfully distinct:

| Operation | Role |
|---|---|
| `stop` | Terminate and commit to the current answer. |
| `continue` | Exploit the current reasoning trajectory by extending it. |
| `branch` | Explore an independent alternative trajectory. |
| `verify_check` | Audit the current candidate answer or reasoning path. |

The implementation must preserve these distinctions. In particular:

- `continue` must not restart from scratch.
- `branch` must not receive the full previous reasoning trace.
- `verify_check` must record whether it used self-checking, deterministic/tool checking, or both.
- `stop` must distinguish committing to an existing answer from eliciting an answer from a partial trace.

## Inputs

Each operation should consume a `ReasoningState` object produced by the state-extraction stage.

The operation may also need:

- the parent `TaskExample`
- parser/scoring utilities
- verifier/checker utilities
- model backend/inference client
- operation prompt templates
- decoding configuration
- token-budget configuration
- run/config metadata

## Outputs

Each operation should return a standardized result object that can later be converted into a `CounterfactualOutcome`.

The result should include at least:

| Field | Description |
|---|---|
| `state_id` | Source reasoning state. |
| `task_id` | Parent task. |
| `operation` | `stop`, `continue`, `branch`, or `verify_check`. |
| `operation_subtype` | More specific subtype if applicable. |
| `rollout_seed` | Generation seed or deterministic marker. |
| `prompt_text` | Prompt sent to model, or reference/hash if too large. |
| `generated_text` | Model output or operation result text. |
| `parsed_answer` | Parsed answer/code/label if available. |
| `tokens_generated` | New tokens generated by the operation. |
| `tool_cost` | Checker/tool cost if applicable. |
| `runtime_seconds` | Operation runtime. |
| `failure_type` | Structured failure type if operation fails. |
| `metadata` | Operation-specific metadata. |

Do not score correctness in this prompt unless it is natural to call the existing scoring interface. The counterfactual collection stage will handle final scoring consistently. However, operations may call checkers when the operation itself requires checking.

## Operation 1: `stop`

The `stop` operation terminates reasoning and commits to an answer.

There are two internal subtypes:

| Subtype | Meaning |
|---|---|
| `stop_commit` | Commit to an already parsed candidate answer. |
| `stop_elicit` | Ask the model to produce a final answer from the partial trace. |

### `stop_commit`

Use `stop_commit` when the state already contains a parsed candidate answer.

Requirements:

- no generation should be needed
- token cost should be zero
- output should commit to the existing candidate
- record that this was a commit, not a generated answer

### `stop_elicit`

Use `stop_elicit` when no candidate answer exists in the state.

The model should be asked to give a final answer based only on the reasoning so far.

Prompt intent:

    Based only on the reasoning so far, give the final answer now.
    End with: Final answer: <answer>

The exact prompt should be stored in a prompt template file or config.

Suggested max tokens:

| Domain | Max tokens |
|---|---:|
| Math | `64` |
| Code | `128` |
| Symbolic | `64` |

The result should record that the answer was elicited from a partial trace.

### Stop Constraints

Do not allow `stop_elicit` to restart full reasoning. It should be a short answer extraction/completion step.

Do not count `stop_commit` as free if any generation or parsing repair is performed. If generation occurs, it is `stop_elicit`.

## Operation 2: `continue`

The `continue` operation extends the current reasoning trajectory.

It receives:

- original task prompt
- current trace prefix
- current candidate answer if present
- no instruction to restart

Prompt intent:

    Continue the reasoning from where it left off. Do not restart or solve from scratch.
    If you reach a conclusion, end with: Final answer: <answer>

Suggested max tokens:

| Domain | Max tokens |
|---|---:|
| Math | `512` |
| Code | `768` |
| Symbolic | `384` |

### Continue Constraints

The operation must include the trace prefix.

It must explicitly discourage restarting from scratch.

It should preserve continuity with the existing reasoning path.

It should record whether the generated continuation:

- produced a new candidate answer
- changed a previous candidate
- repeated prior reasoning
- reached a final answer marker
- remained incomplete

Do not implement branch behavior under `continue`.

## Operation 3: `branch`

The `branch` operation starts an independent alternative trajectory.

It receives:

- original task prompt
- optionally the current candidate answer
- not the full previous reasoning trace

Prompt intent:

    Solve the problem independently from scratch. Do not rely on the previous reasoning path.
    Use a different approach if possible.
    End with: Final answer: <answer>

If the current state has a candidate answer, the branch prompt may include:

    A previous attempt suggested this candidate answer: <candidate>. You may disagree with it.

The exact templates should be stored in config/prompt files.

Suggested max tokens:

| Domain | Max tokens |
|---|---:|
| Math | `768` |
| Code | `1024` |
| Symbolic | `512` |

### Branch Constraints

This is a critical methodological invariant:

> `branch` must not receive the full prior trace.

The branch operation should not see the previous reasoning path. It may see the original prompt and optionally the current candidate answer. This prevents branch from becoming contaminated continuation.

Record metadata indicating whether a previous candidate was included.

The branch result should record:

- branch output
- parsed branch candidate if available
- whether it agrees with the original candidate if one exists
- branch token cost
- branch failure type if any

## Operation 4: `verify_check`

The `verify_check` operation audits the current candidate answer or reasoning path.

This operation has three reporting variants:

| Variant | Meaning |
|---|---|
| `self_check` | Model-only verification or critique. |
| `tool_check` | Deterministic verifier/test/checker only. |
| `full_check` | Combined model audit plus deterministic/tool feedback where available. |

The operation implementation should make these subtypes explicit in outputs.

The main pilot may use `full_check`, but the code should allow reporting or running `self_check` and `tool_check` variants separately where feasible.

## Verify/Check Behavior by Domain

### Math

For math, verify/check may include:

- model self-checking
- recomputation of fragile steps
- deterministic answer equivalence after a revised answer is produced
- symbolic/numeric checking where applicable

Prompt intent:

    Check the current solution for errors. Recompute any fragile steps.
    If the current answer is correct, keep it. If it is wrong, correct it.
    End with: Final answer: <answer>

Suggested max tokens:

| Domain | Max tokens |
|---|---:|
| Math | `384` |

### Code

For code, verify/check should use executable tests when available.

Expected behavior:

1. extract current code candidate if present
2. run relevant tests/checks
3. if tests pass, return a successful check result
4. if tests fail, produce a compact failure summary
5. allow one correction pass
6. run tests again where applicable
7. record all checker metadata

Prompt intent after failure:

    The current code failed the following checks:
    <failure_summary>

    Fix the issue. Return the final corrected code in one code block.

Suggested max tokens:

| Domain | Max tokens |
|---|---:|
| Code | `768` |

Only allow one correction pass per verify/check operation. More correction turns would turn this into an agentic debugging loop and should not be part of the pilot.

### Symbolic

For symbolic tasks, verify/check may include:

- deterministic label validation
- rule/fact consistency check
- model self-checking
- one correction pass if applicable

Prompt intent:

    Check whether the current answer follows from the facts and rules.
    If the answer is unsupported or contradicted, correct it.
    End with: Final answer: <answer>

Suggested max tokens:

| Domain | Max tokens |
|---|---:|
| Symbolic | `384` |

## Verify/Check Constraints

Do not treat verifier/checker calls as free.

Record:

- checker calls
- tests run
- execution time
- failure summaries
- whether correction was attempted
- whether correction changed the answer
- whether deterministic checking was available
- subtype: `self_check`, `tool_check`, or `full_check`

Do not expose large raw logs in model prompts. Use compact failure summaries.

Do not allow unlimited repair loops.

Do not give CVRO verifier access that later baselines cannot also use.

## Operation Prompt Templates

Prompt templates should live in files or config entries, not hardcoded deep in operation logic.

Suggested organization:

| Template | Purpose |
|---|---|
| `stop_elicit` | Short final-answer elicitation from partial trace. |
| `continue` | Extend current trajectory. |
| `branch` | Independent alternative trajectory. |
| `verify_math` | Math self-check/correction. |
| `verify_code_failure` | Code correction after failed tests. |
| `verify_symbolic` | Symbolic consistency check. |

The template system should support injecting:

- original task prompt
- trace prefix where allowed
- current candidate where allowed
- failure summary where applicable
- answer-format instruction

## Decoding Configuration

Use configurable decoding settings per operation.

Suggested defaults:

| Operation | Temperature | top_p |
|---|---:|---:|
| `stop` | `0.0` or `0.2` |
| `continue` | `0.7` |
| `branch` | `0.8` |
| `verify_check` | `0.2` |

These should not be hardcoded. Store them in config.

Branch should generally be more diverse than continue. Verify/check should generally be low-temperature.

## Token Budget Configuration

Use configurable max-token budgets.

Suggested defaults:

| Operation | Math | Code | Symbolic |
|---|---:|---:|---:|
| `stop_elicit` | `64` | `128` | `64` |
| `continue` | `512` | `768` | `384` |
| `branch` | `768` | `1024` | `512` |
| `verify_check` | `384` | `768` | `384` |

The operation should record if output was truncated by the token limit.

## Operation Failure Types

Use structured failure types.

Examples:

| Failure type | Meaning |
|---|---|
| `model_error` | Model call failed. |
| `timeout` | Operation exceeded timeout. |
| `parse_error` | No usable output could be parsed. |
| `checker_error` | Deterministic checker failed unexpectedly. |
| `no_candidate` | Operation required a candidate but none was available. |
| `invalid_state` | State missing required fields. |
| `unsafe_code` | Code violated sandbox policy. |
| `budget_exceeded` | Operation exceeded allowed budget. |
| `unknown_error` | Unexpected failure. |

Do not crash the whole pipeline for individual operation failures. Return a structured failure result.

## Local Dry Run

Support local dry runs for operation construction and execution.

A dry run should be able to:

- load a tiny set of states
- construct prompts for all four operations
- optionally use a mock model backend
- return schema-valid operation results
- exercise parser/checker integration for verify/check when possible

This should work before large model inference is available.

## Tests and Validation Checks

Add tests or validation scripts for:

### Stop

- uses `stop_commit` when candidate exists
- uses `stop_elicit` when no candidate exists
- `stop_commit` has zero generation cost
- `stop_elicit` uses short max-token budget

### Continue

- includes trace prefix
- does not use branch prompt
- preserves original prompt and partial trace
- records continuation metadata

### Branch

- does not include full prior trace
- may include current candidate only if configured
- records candidate agreement where available
- uses branch-specific decoding settings

### Verify/Check

- records subtype
- uses one correction pass maximum
- records checker cost
- handles no-candidate cases
- produces compact failure summary for code
- does not treat checker/tool calls as free

### Cross-Operation

- all operations return standardized result objects
- operation results include state ID, task ID, operation, subtype, cost, metadata
- operation prompts are versioned or hashable
- failures are structured
- local dry run passes

## Integration With Later Counterfactual Collection

The counterfactual collection stage will call each operation from the same saved state.

Make sure the operation functions/classes can be called independently and repeatedly with different rollout seeds.

The design should support:

- `1` stop rollout
- `2` or `3` continue rollouts
- `2` or `3` branch rollouts
- `1` verify/check rollout

Do not assume a fixed number of rollouts inside the operation itself. That is controlled by the counterfactual collection stage.

## Do Not Do

Do not implement full counterfactual collection here.

Do not compute final action values here.

Do not train controllers here.

Do not run Best-of-N baselines here.

Do not implement an unrestricted multi-turn agent.

Do not let branch see the full prior trace.

Do not let continue restart from scratch.

Do not silently ignore checker/tool cost.

Do not manually repair model outputs.

Do not add operations beyond `stop`, `continue`, `branch`, and `verify_check`.

## Definition of Done

This prompt is complete when the repository can:

1. load saved `ReasoningState` objects,
2. construct prompts for stop, continue, branch, and verify/check,
3. execute or mock each operation,
4. return standardized operation result objects,
5. enforce branch/continue/verify methodological constraints,
6. record costs, subtypes, and failures,
7. run operation-level tests,
8. support later counterfactual collection from the same state.

The operation implementation must be clean, reusable, and ready for the next stage: forcing all four operations from each saved state to collect counterfactual outcomes.
